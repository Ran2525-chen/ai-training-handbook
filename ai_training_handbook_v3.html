PART 01 · 配置

四条接入路径：按需选择，可以并行

不同路径适合不同角色与需求，实际使用中可同时开启多条。建议所有成员先配置 ChatGPT 账号路径，数据分析同事额外配置 API Key 或 Codex CLI。

路径适合人群典型场景技术门槛
ChatGPT 账号所有成员文档处理、资料阅读、报告撰写、PPT 大纲⭐ 无需配置
OpenAI API Key数据分析、有自动化需求批量处理 Excel、生成脚本、调用模型 API⭐⭐ 需申请 Key
Codex CLI / VS Code需要跑 Python、数据脚本数据清洗、图表生成、批量文件转换⭐⭐⭐ 需安装环境
国内兼容 API访问受限或成本敏感千问、DeepSeek、Kimi 等国产模型替代⭐⭐ 替换三个参数
建议顺序：先完成 ChatGPT 账号配置 → 数据分析同事再加 API Key → 需要本地跑脚本再装 Codex CLI → 如遇访问问题再切国内 API。
PART 01 · 配置 — ChatGPT 账号

路径一：ChatGPT 账号 SOP

适合所有项目成员。用于日常文档处理、资料阅读、会议纪要、访谈整理、报告润色、PPT 大纲生成。ChatGPT Plus（$20/月）解锁 Projects、Deep Research、Data Analysis 全部能力。

1
注册并升级 Plus
访问 chat.openai.com，注册账号后升级 Plus 计划。升级后可使用 GPT-4o、Projects、Deep Research、Data Analysis、Tasks 全部功能。
2
按项目建立 Project
在左侧边栏点击「+ New project」，按项目名称创建，例如「四川餐饮个体工商户动态监测」。每个项目独立管理文件和对话，不同项目上下文隔离。
3
上传核心项目文件
在 Project 内上传：合同/方案、问卷模板、访谈记录、报告模板、会议纪要。建议文件命名规范，如「2024-访谈记录-成都-商户A.pdf」，便于 AI 引用时定位。
4
写入项目级指令（Custom Instructions）
在 Project Settings 中设置：
• 报告风格：「使用政府咨询报告风格，客观陈述，避免绝对化表述」
• 客户口径：「甲方为某省某厅，正式场合称"研究团队"」
• 禁止事项：「不得编造数据来源，不得使用未经核实的统计数字」
• 常用格式：「所有报告段落使用二级标题，要点用短句，数据需标注来源」
5
建立固定对话入口
在 Project 内为常用任务建立专属对话并命名，避免混用：
• 「📄 资料整理与摘要」——访谈、政策文件阅读
• 「📝 报告撰写」——初稿、润色、修改
• 「📊 数据分析」——上传文件、生成表格和图表
• 「🖥 PPT 大纲」——结构生成和讲稿
关键原则：同一项目的所有 AI 对话都在对应 Project 内进行，避免上下文丢失、重复解释背景。
PART 01 · 配置 — API Key

路径二：OpenAI API Key SOP

适合数据分析同事和有批量处理、自动化需求的成员。API 路径解锁了脚本调用、批量处理和与 Codex 联动的能力，但需要额外的安全注意事项。

1
申请 API Key
访问 platform.openai.com → API Keys → Create new secret key。命名建议加前缀，如「proj-sichuan-analysis」，便于区分项目用途。Key 只显示一次，立即保存。
2
安全存储——必须用环境变量
绝对禁止将 Key 写入代码文件或上传到任何平台。正确做法：
macOS / Linux：在终端执行
echo 'export OPENAI_API_KEY="sk-..."' >> ~/.zshrc && source ~/.zshrc Windows：系统设置 → 环境变量 → 新建 OPENAI_API_KEY 代码中调用：import os; api_key = os.environ["OPENAI_API_KEY"]
3
设置预算上限与用量提醒
在 platform.openai.com → Billing → Usage limits 中设置月度硬上限（如 $30）和软提醒（如 $20 时邮件通知）。测试时先用小样本数据（10-20 行）验证效果，再扩量。
4
选择合适的模型
• gpt-4o：综合能力最强，适合复杂分析与报告生成
• gpt-4o-mini：成本低 20 倍，适合简单分类、批量摘要
• 建议：先用 mini 跑通流程，正式交付时切换 4o 做质量保障
⚠️ 安全提醒：API Key 等同于账号密码。一旦泄露立即在 platform.openai.com 撤销（Revoke）并重新生成。不要将含 Key 的代码上传 GitHub 或飞书公开文件夹。

PART 01 · 配置 — 国内兼容 API

路径四：国内兼容 API SOP

适合访问受限、成本敏感或需要国产模型备选的场景。核心逻辑：国内主流大模型均提供 OpenAI-compatible 接口——只需替换三个参数，无需改动其余代码。

🔧 三个核心参数
import openai
client = openai.OpenAI(
  base_url="https://api.xxx.com/v1", # 服务商地址
  api_key="sk-xxxxx", # 服务商 Key
)
response = client.chat.completions.create(
  model="qwen-plus", # 服务商模型名
  messages=[...]
)
📋 主流服务商参数对照
服务商base_urlmodel 示例
阿里千问dashscope.aliyuncs.com/compatible-mode/v1qwen-plus
DeepSeekapi.deepseek.com/v1deepseek-chat
Kimiapi.moonshot.cn/v1moonshot-v1-8k
字节豆包ark.cn-beijing.volces.com/api/v3ep-xxxxxx
1
验证输出质量
用同一套提示词对比 OpenAI 和国内模型的输出，重点检查逻辑完整性、数据准确性、格式一致性。
2
核算 Token 成本
各服务商 Token 定价差异较大。测试时记录每次 prompt_tokens 和 completion_tokens，换算实际成本后再决定大规模使用。
3
确认数据安全要求
项目涉及客户资料时，必须先确认：①服务商数据是否出境；②是否用于模型训练；③是否签署数据安全协议。详见合规边界章节。
4
配置超时与重试
国内服务网络延迟较高，建议代码中设置 timeout=60, max_retries=3，避免因网络抖动导致脚本中断。

PART 02 · ChatGPT — Projects

Projects：项目知识库与长期上下文管理

Projects 把文件、指令和对话集中在同一个项目上下文中，让 AI 始终"记得"你的项目背景——不再每次重新解释客户是谁、报告风格是什么、禁止事项有哪些。

✅ 最适合的场景
• 政府咨询项目——周期长、资料多、多版本报告
• 长期访谈研究——多轮访谈记录归档与交叉分析
• 多人协作项目——统一口径、格式和客户要求
• 周报 / 月报——每周基于同一背景生成进展更新
📁 推荐的文件组织方式
• 合同 / 项目方案（确定范围和口径）
• 访谈提纲 + 访谈记录（按日期命名）
• 政策文件（按政策名称 + 发布年份命名）
• 报告模板（确定格式标准）
• 客户反馈（历次沟通纪要）
📋 Custom Instructions 写法示例（点击展开） ▶
你是本项目的研究助手。项目背景：[四川省餐饮个体工商户动态监测研究]，委托方为[某省商务厅]。

输出风格要求：
- 使用政府咨询报告风格，客观陈述，避免绝对化表述
- 政策数据必须标注来源和时间，不得编造统计数字
- 报告段落使用二级标题，要点用短句，避免过度修辞

禁止事项：
- 不得将客户名称、项目编号写入任何输出
- 不得引用未经我提供的"内部数据"或"调研结论"
- 所有政策摘引须注明"来源：[文件名]"

常用格式：数据分析结论用"表 + 文字说明"格式；报告摘要不超过 300 字。
注意：Projects 内上传的文件会在对话中被 AI 读取，请勿上传含客户个人隐私、涉密数据或合同金额的原始文件。上传前应脱敏处理或仅上传摘要版本。
PART 02 · ChatGPT — Deep Research

Deep Research：带来源的深度研究

Deep Research 可结合网页、上传文件和已启用应用，生成带来源引用的研究报告。适合行业研究、政策梳理、竞品分析、区域对标——它会自主规划研究路径，多步检索后汇总。

✅ 适用场景
• 政策梳理：检索某领域近三年政策演变
• 行业数据：汇总行业规模、增速、区域分布
• 竞品/对标：梳理其他地区典型案例和做法
• 证据收集：整理可写入报告的可引用事实
⚠️ 使用要求
• 必须要求"列出来源 URL 和发布时间"
• 区分"事实陈述"和"AI 判断/推断"
• 关键数据落库前必须点进原始链接核实
• 不得将未核实的 Deep Research 结论直接写进报告
📝 Deep Research 提示词模板（点击展开） ▶
请围绕【四川省餐饮个体工商户参与新消费场景】，执行以下研究任务：

1. 检索近三年（2022-2024）相关政策文件，列出政策名称、发布机关、发布时间
2. 汇总行业规模数据（市场规模、商户数量、增速），标注数据来源和统计口径
3. 整理四川省内典型案例或示范场景（不少于3个），注明地区和来源
4. 输出格式：
- 研究摘要（200字以内）
- 证据表（来源 | 核心信息 | 时间 | URL）
- 可写入报告的核心观点（3-5条，区分"事实"和"判断"）

注意：所有信息必须标注来源，不确定的内容请注明"待核实"。
使用逻辑：Deep Research 负责"广度检索 + 结构化汇总"，人工负责"核实关键数据 + 判断是否可用"。AI 给你省去搜索和整理时间，核实工作仍需人来做。
PART 02 · ChatGPT — 文件与数据分析

Files + Data Analysis：文件阅读与数据分析

Data Analysis 支持上传 Excel、PDF、Word、CSV、图片等格式，AI 可直接读取并执行分析——生成统计表、交叉分析、图表和分析结论，无需提前写脚本。

📊 Excel / 问卷数据清洗典型流程（点击展开） ▶
1
上传文件 + 说明数据背景
「这是一份问卷星导出的 Excel，第一行是表头，A 列是受访者编号，B-F 列是李克特量表（1-5 分），G-H 列是开放问答。」
2
要求先做数据质检
「请先检查：① 有多少行有效数据，② 有没有完全空白行，③ 量表列有没有超出 1-5 范围的异常值，④ 有没有重复受访者编号。」
3
分组统计与交叉分析
「按受访者行业分组，计算每组的 B-F 列均值，输出为表格，并说明各组之间差异是否显著。」
4
生成图表
「基于上面的分组统计结果，生成一张分组对比柱状图，颜色方案用蓝绿系，标注每组样本量 n=xx。」
5
输出分析结论
「综合以上数据，用 3-5 句话描述核心发现，区分'数据事实'和'可能的解释'，避免过度推论。」
📄 PDF / Word 文件阅读典型任务（点击展开） ▶
政策文件抽取
上传文件后说：
「请从这份政策文件中抽取：
① 政策名称和发布时间
② 适用对象
③ 核心支持措施（用表格列出：措施 | 支持力度 | 申报条件）
④ 执行部门和联系方式
输出为结构化表格。」
访谈记录整理
上传访谈记录后说：
「请整理这份访谈记录，输出：
① 受访者基本信息（行业/规模/地区）
② 核心观点（分主题归纳，3-5个主题）
③ 关键引用句（直接引用原文，标注大概位置）
④ 待核实或矛盾信息」
数据安全提醒：上传前请确认文件不含客户姓名、手机号、身份证号等个人敏感信息。问卷数据建议先在本地删除个人标识列再上传。
PART 02 · ChatGPT — Tasks

Tasks：定时提醒与动态监测

Tasks 支持三类任务：一次性提醒、周期性提醒、监测型任务（基于搜索结果触发）。适合项目节点提醒、政策动态跟踪、周报生成等周期性工作。

任务类型配置方式项目典型用法
一次性提醒指定日期和时间「2024-11-30 提醒我提交项目中期报告」
周期性提醒每天 / 每周 / 每月「每周五下午 3 点，提醒整理本周项目进展要点」
监测型任务基于搜索关键词「每天检查"四川餐饮扶持政策"相关新闻，有新内容时通知我」
💡 Tasks 设置示例（点击展开） ▶
# 周报生成提醒
「每周五下午 4 点，提醒我：
根据本周会议纪要和进展，生成一份项目周报草稿，
格式：本周完成 / 下周计划 / 待确认事项 / 风险提示。」

# 政策监测任务
「每天早上 9 点，搜索昨日关于'个体工商户'+'扶持'+'四川'的政策新闻，
如有新内容，汇总标题、来源、发布时间发给我。」
使用建议：Tasks 适合"提醒触发"场景，不适合替代项目管理工具（飞书、Teambition）。把 Tasks 理解为"AI 助手会主动提醒你做的事"，而不是项目进度跟踪系统。
PART 02 · ChatGPT — 写作与 Canvas

写作与 Canvas：报告初稿与多轮修改

Canvas 是专为长文档多轮修改设计的写作界面——你可以在同一画布中对特定段落进行局部修改，同时保持其他内容不变，避免每次修改都要重新生成全文。

适合用 Canvas 的场景
• 报告正文多轮修改（5000字以上）
• 访谈摘要逐段打磨
• PPT 讲稿与正文文档同步修改
• 同一报告不同章节风格统一
写作质量控制要点
• 政策数据、金额、排名、时间口径——人工复核
• AI 给的"研究发现"——必须有原始数据支撑
• 报告结论——不得由 AI 单独生成，需基于真实访谈/数据
• 客户专有术语——建议在 Custom Instructions 中列出
📝 报告初稿生成流程（点击展开） ▶
1
提供结构框架
先给 AI 报告大纲（可来自 PPT 或项目方案），让它理解整体逻辑再开始写。
2
分章节生成，不要一次全写
「先帮我生成第二章'现状分析'，基于以下数据点：...」每章单独生成、单独检查。
3
用 Canvas 局部修改
在 Canvas 中选中某一段，右键选择「修改这一段」，说明修改意图：「这段太学术化，改成更口语化的政府报告风格。」
4
统一风格检查
全文生成后，让 AI 做「风格一致性检查」：「检查全文是否有不一致的表达方式，例如同一指标用了不同称呼。」
核心原则：AI 不能替代专业判断。所有政策数据、金额、排名、时间口径必须人工复核。报告中的研究结论必须基于真实的访谈记录或数据，AI 只辅助表达，不生产结论。
PART 02 · Codex — CLI 与 IDE

Codex CLI / IDE：把意图变成可执行代码

Codex CLI 可以读取本地目录内的所有文件，理解项目结构，然后根据自然语言描述生成、修改并执行代码。IDE 扩展则在写代码时提供行内实时补全与解释。

Codex CLI 核心能力
• 读取本地目录内任意格式文件
• 理解文件间关系（如多张 Excel 之间的关联）
• 生成 Python / Bash / SQL 脚本
• 自动执行代码并返回结果
• 支持 Record & Replay 把操作录制为 Skill
VS Code 扩展核心能力
• 行内代码补全（Tab 接受建议）
• 选中代码后解释或重写（右键菜单）
• 在侧边栏对话修改当前文件
• 支持 Python、VBA、HTML、SQL、PPT 自动化脚本
🔄 Codex 运行 / 验收 / 回滚三步法（点击展开） ▶
运行
先用小样本测试
正式运行前先取 10-20 行数据测试脚本逻辑，确认输出格式和计算结果符合预期。
验收
三项必检
① 行数是否与原数据一致（没有莫名丢失数据）；② 数值范围是否合理（如分值应在 1-5 之间）；③ 输出文件能否正常打开、格式是否正确。
回滚
保留原始文件
脚本运行前务必备份原始数据文件。建议命名规范：data_原始.xlsx → 脚本生成 → data_清洗后_20241130.xlsx。发现问题时可立即回到原始文件重跑。
🎬 Record & Replay：把演示录成可复用 Skill（点击展开） ▶
1
定义目标
明确录制范围：「每周从系统导出报表 → 去重清洗 → 生成分组统计表」。边界越清晰，录制质量越高。
2
在 macOS Codex App 录制
开启录制后完整执行一遍操作，Codex 自动捕获每一步——打开文件、操作数据、导出结果。
3
生成 Skill
录制完成后 Codex 生成结构化 Skill，包含步骤、可修改参数（如日期范围）和异常处理逻辑。
4
复用执行
下次执行只需调用 Skill，修改参数（如本周日期范围）后运行，无需重复手动操作。适合周度例行任务。
适用场景：周期性报表导出、标准化文件整理、多系统间重复数据搬运。界面大幅变动后需重新录制。

AGENTS.md 与 Skills：团队规则与可复用能力

AGENTS.md 是项目目录中的规则文件，告诉 Codex 这个项目的数据口径、命名规范和禁止事项；Skills 则把验证过的操作流程封装成可重复调用的能力模块。

📋 AGENTS.md 模板（点击展开） ▶
# AGENTS.md — 四川餐饮个体工商户动态监测项目

## 项目背景
本项目研究四川省餐饮类个体工商户的经营状态与政策支持情况。
数据主要来源：问卷调查（n=312）、政策文件、统计年鉴。

## 数据口径
- 个体工商户：指在工商部门登记、非法人主体的经营户
- 餐饮类：仅包含 GB/T 4754-2017 中 C 大类餐饮服务
- 时间口径：2021-2023 年数据，2024 年为预估值需标注

## 文件命名规范
- 原始数据：data_原始_[来源]_[日期].xlsx
- 清洗后数据：data_清洗_[版本号]_[日期].xlsx
- 分析结果：output_[分析类型]_[日期].xlsx/png

## 禁止事项
- 禁止编造任何统计数据或政策内容
- 禁止将原始问卷中的个人信息（姓名、手机）写入输出文件
- 禁止在没有数据支撑的情况下生成"研究结论"
- 禁止删除原始数据文件（只读，不修改）

## 输出格式
- 统计表：xlsx，含数据来源注释行
- 图表：png，300dpi，含标题、轴标签、数据标注
- 代码：.py 文件，含注释说明每个步骤的目的
⚙️ Skills 适合封装的场景
• 问卷数据质检（缺失值 / 异常值检查）
• 标准分组统计（按行业、地区、规模）
• 热力图生成（可配置参数化颜色和维度）
• Word 报告批量格式化
• 多文件合并去重
🔌 Plugins / Subagents
Plugins 连接外部工具（飞书、Jira、Slack），适合有 IT 支持的团队；

Subagents 让 Codex 把任务分解后并行执行——例如同时处理 10 份访谈记录的摘要提取，而不是一份一份来。

PART 02 · 提示词框架

六要素提示词框架：一次说清，减少来回

好的提示词不靠灵感，靠结构。六要素框架适用于所有场景——报告写作、数据分析、政策检索，进入 Part 03 实操前先掌握这个基础工具。

六要素结构
角色
「你是一位政府咨询顾问，擅长数据分析和政策研究」
任务
具体要做什么：「整理以下访谈记录中的核心观点」
背景
项目上下文、数据说明、受众是谁
步骤
如有顺序要求，逐步说明：「先检查，再统计，再生成图表」
格式
期望输出格式：「Markdown 表格」「Word 段落」「Python 脚本」
示例
给出参考示例——有例子比无例子效果好 50%+
数据分析提示词示例
【角色】你是数据分析专家
【任务】清洗并分析上传的问卷数据
【背景】312份餐饮个体工商户问卷，
按行业子类分组（A列），
B-F列为5分制量表
【步骤】
1. 先做数据质检（空值/异常值/重复）
2. 按A列分组计算B-F均值
3. 生成分组对比图表
4. 输出3-5句分析结论
【格式】Excel表格 + PNG图表 + 文字说明
【示例】[附上上次分析的截图或格式]
六要素不必每次都写全，但任务 + 背景 + 格式三要素是最低要求。缺少背景是提示词失效最常见的原因。

合规与边界

合规使用规范：四条边界线

AI 工具的使用必须在合规边界内进行。以下四条规则适用于所有项目成员，是使用 ChatGPT 和 Codex 的前提条件——不确定的情况下，先问项目负责人再操作。

🚫 客户资料与涉密数据
  • 禁止将客户全名、手机号、身份证号、合同金额上传至任何 AI 平台
  • 问卷数据上传前必须删除个人标识列（姓名/电话/地址）
  • 标注"内部资料"或"保密"的文件禁止上传
  • 政府项目涉及的未公开数据（如内部统计数据）禁止上传
⚠️ 上传文件安全规则
  • 每次上传前问自己："如果这个文件被公开，会有什么后果？"
  • 优先上传摘要版本而非完整原始文件
  • API Key 文件、数据库配置文件禁止上传
  • 国内模型使用前确认服务商的数据出境与训练协议
🚫 AI 幻觉核验要求
  • 所有 AI 生成的政策数据必须点进原始链接核实
  • 统计数字（规模、增速、排名）必须标注来源后人工确认
  • AI 生成的"引用"需逐一核实是否存在原始出处
  • Deep Research 结论不得直接用于正式报告，需二次核实
✅ 交付物复核规则
  • 所有含数据的交付物，须有人工复核记录（谁 · 何时 · 核实了什么）
  • AI 生成的报告初稿必须经项目负责人审阅后方可发送客户
  • 代码生成的 Excel / 图表，需与原始数据抽样比对验证
  • 政策建议类内容，须咨询专业判断后写入，不可直接引用 AI 结论
总原则：AI 是你的助手，不是背书人。所有对外交付物的准确性、合规性和专业性，最终责任仍在项目团队。
不确定时的原则：宁可多花时间手动核实，也不要用未经验证的 AI 输出直接交付。
总结

最后带走 4 个判断

不必把每个功能都用上。这 4 个判断帮你在面对新任务时，快速决定该用 AI 的哪个能力来应对。

01
ChatGPT 先做协作与拆解，Codex 再做执行与自动化
遇到模糊的大任务，先用 ChatGPT 把问题拆清楚、材料整理好，再交给 Codex 生成脚本执行。两者是上下游关系，不是竞争关系。
02
国内模型先按 OpenAI-compatible 方案验证，通过后再推广
不需要重学 SDK，只需替换三个参数。验证顺序：质量对比 → 成本核算 → 数据安全确认 → 推团队使用。
03
高质量提示词 = 角色 + 任务 + 背景 + 步骤 + 格式 + 示例
六要素框架通用于报告写作、数据分析、政策检索、PPT 生成。最低要求：任务 + 背景 + 格式，缺背景是提示词失效最常见的原因。
04
所有对外交付物必须人工复核，AI 不背书，责任在团队
AI 负责提速、AI 负责初稿、AI 负责格式化——但数据准确性、结论合理性、合规边界，永远是项目团队的责任。
